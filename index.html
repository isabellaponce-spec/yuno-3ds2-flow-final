3DS2 Server API — Final Flow for Zuora (Option 1)

Goal: Enable Zuora to run 3DS2 from their backend (HPM) for the merchants that do not use Yuno's SDK. All 3DS context travels inline on the existing POST /v1/payments request — the same pattern Yuno already uses with fraud providers and with the Direct workflow today.

Decision (15 May 2026): Option 1 — single call. The device fingerprint, browser data and cardholder info travel inline on the V1 Payments request. Confirmed by Jairo in #yuno-3ds: "it needs to be the 1, like with fraud providers."
Grounded in existing Yuno docs. This proposal does not introduce new top-level endpoints. It reuses the existing Direct workflow + the existing three_d_secure response object in payment_method.detail.card.card_data. References:
  • Basic concepts — 3DS
  • Security & compliance — 3D Secure
  • Direct integration — 3DS configuration & testing

2. The two scenarios (Paloma's framing, mapped to existing capabilities)

Scenario A — Gateway returns a Redirect URL (Stripe, Adyen, the majority)
Fully covered by the existing Direct workflow. Zuora calls POST /v1/payments, Yuno returns the gateway's redirect_url in payment.payment_method.payment_method_detail.card, Zuora redirects the cardholder, the gateway closes the challenge, the result flows back via webhook with the populated three_d_secure response object. No new build. Parameter-mapping work only (per Paloma).
Scenario B — Gateway does not return a Redirect URL (server-to-server gateways, e.g. Cardinal Commerce / CyberSource)
Zuora sends device fingerprint + browser data + cardholder info inline on the V1 Payments request, inside a new three_d_secure request object. Yuno runs the 3DS authentication server-to-server via Netcetera and either (a) returns the auth outcome inline with the existing three_d_secure response fields, or (b) returns a redirect_url for the ACS challenge using the exact same Direct-workflow pattern. Net-new: the request-side three_d_secure object and the s2s authentication wiring inside Yuno.
Provider list update (14 May call): Guoqing clarified that the gateway list per pattern does not change the build, because Zuora always embeds the PSP's own JS and the handoff to Yuno is identical regardless of gateway. Jairo and Bernabe accepted in real time. We proceed without waiting for the list, scoping against the gateways already in the tracker. The list remains available if needed for edge cases.

5. The endpoint surface

POST /v1/payments (existing, request extended for Scenario B)

Triggered by: Zuora at submit. Single call carries payment_method (token or card), amount, currency, plus the new optional request-side three_d_secure object containing browser data + device fingerprint + cardholder info.

Returns (Scenario A and B): existing response shape. Either SUCCEEDED with the populated three_d_secure response object when frictionless, or PENDING / WAITING_ADDITIONAL_STEP with a redirect_url in payment.payment_method.payment_method_detail.card when a challenge is required.

Pattern match: identical to how fraud providers receive their context inline on V1 Payments today. No new top-level endpoint. Only a new optional object on the existing payment request, plus s2s authentication wiring on Scenario B.

GET /v1/payments/{id} (existing, no change)

Used by Zuora as fallback to retrieve the final payment status if a webhook is missed.

Webhook (existing, no change)

Delivers the 3DS challenge outcome and final payment status to Zuora's registered webhook endpoint. Status transitions: CREATED → PENDING → IN_PROCESS → SUCCEEDED / DECLINED / ERROR.

7. MIT and portability post-migration (Jarrett)

QuestionAnswer
Do we need to migrate ECI / CAVV / dsTransID from Zuora's vault? No. Schemes rely on the scheme-level NTI from the original CIT. Yuno does not need to store the 3DS response for future use.
Are MITs exempt from 3DS post-migration? Yes, in EU, as long as Yuno passes the proper stored credential / scheme-level NTI from the initial CIT.
Can Yuno orchestrate MITs across PSPs? Yes — only if Yuno has the scheme-level NTI (not PSP-level).
What about the 7 PSPs returning obfuscated NTIs? Yuno can still send MITs back to the same PSP that issued the obfuscated value. Cross-PSP orchestration is blocked until those PSPs release the raw NTI. Zuora needs to de-obfuscate before migration — out of Yuno's scope, already escalated in the daily.

8. Hard constraints (Zuora's red lines)

RequirementWhy it matters
No Yuno SDK (foreground or background)Zuora HPM already has its own browser collection layer; injecting our SDK breaks their tenant model and PCI scope.
No redirect to a Yuno domain for data collectionThe only acceptable redirect is the ACS challenge itself, hosted by the issuer. Yuno does not host any UI in this flow.
Pure server-to-server (Scenario B)Their backend calls our backend, period. Same pattern they use today with Cardinal Commerce.
PCI scopeYuno's Direct workflow is gated on PCI-compliant merchants per the existing docs — Zuora qualifies.
